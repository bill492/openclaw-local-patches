# OV-011

## Purpose

OV-011 is the QMD session-export race patch for OpenClaw `2026.4.14`.

Use this when all of the following are true:

- `memory.backend` is `qmd`
- QMD session export is enabled
- the install is vulnerable to duplicate full-manager creation or overlapping session export/update work
- you want the PR `#65226` behavior on a dist install before it is present in the shipped release

This document is the reusable operator reference. Another Codex session should be able to read only this file and implement the same patch on another OpenClaw node.

## What It Fixes

OV-011 has two coordinated runtime parts.

### Part 1: Deduplicate only full QMD manager creation

Without this patch, `getMemorySearchManager()` can miss cache on two concurrent `:full` requests for the same agent and let both calls proceed into `QmdMemoryManager.create(...)`.

That can arm duplicate QMD managers for the same agent and same `qmdDir`.

The final PR fix does not dedupe the entire "QMD-or-builtin manager acquisition" flow. It dedupes only the full QMD create path itself:

- cached full-manager hits still return immediately
- borrowed status-manager hits still return immediately
- only uncached full QMD creation is shared through the pending promise map
- builtin fallback remains outside that pending map

That narrower shape matters because it avoids unnecessary work on the hot cached path and avoids over-keying the pending map by unrelated raw config.

### Part 2: Serialize the session export + QMD update critical section, and let closed waiters drop out

Without this patch, each `QmdMemoryManager` instance only serializes itself with instance-local `pendingUpdate` / `queuedForcedUpdate`.

That is not enough if duplicate managers exist, because they can still both run:

- `exportSessions()`
- `runQmdUpdateWithRetry()`

against the same `qmdDir`.

The fix adds a process-global queue keyed by `this.qmdDir` and wraps only:

- `exportSessions()`
- `runQmdUpdateWithRetry()`
- `this.dirty = false`

inside that queue.

The final PR also makes this queue close-aware:

- `close()` resolves a `closeSignal`
- a manager waiting behind another manager on the shared update queue can exit promptly when closed
- embed work and post-update bookkeeping still remain outside the queue

## Why This Exists

This patch is not generic QMD tuning. It fixes a specific race and its shutdown follow-up:

- duplicate in-flight full managers can be created for the same agent
- those managers can overlap on session export and update
- overlapping writes to exported session markdown can trip OpenClaw's `fs-safe` inode verification
- once a shared update queue exists, a manager closed while waiting behind another manager should not have to sit through that other manager's full update timeout window
- the visible symptom is intermittent `SafeOpenError` / failed QMD update cycles when session export is enabled, plus avoidable slow shutdown when queued managers are closed

## Exact Behavior Change

### Part 1 behavior

Before:

- cache only stored completed managers
- a same-agent full cache miss went straight into `QmdMemoryManager.create(...)`
- concurrent callers could create multiple full managers

After:

- cache store also carries `pendingQmdManagerCreates`
- cached full-manager and borrowed status-manager returns happen before any pending-create-key work
- same-agent full callers check the pending promise map only after those fast paths miss
- the pending map dedupes only `createQmdManager("full")`
- the pending key is derived from the effective QMD cache inputs plus `workspaceDir`
- unrelated raw config differences do not create different pending keys unless they change the effective QMD inputs or workspace
- one QMD create path runs, later callers await the same promise
- if QMD creation fails or returns no manager, callers fall back through the builtin path outside the pending map
- cleanup clears the pending map in `closeAllMemorySearchManagers()`

### Part 2 behavior

Before:

- `runUpdate()` used only instance-local guards
- two managers for the same `qmdDir` could still overlap on `exportSessions()` and `runQmdUpdateWithRetry()`

After:

- queue state is stored under `QMD_UPDATE_QUEUE_KEY`
- queue entries are keyed by `this.qmdDir`
- `runUpdate()` wraps only the export + update critical section in `withQmdUpdateQueue(...)`
- `close()` resolves a `closeSignal`
- `withQmdUpdateQueue(...)` races the prior waiter against `closeSignal`
- a closed manager waiting on the shared queue can return without running export/update work
- embed and post-update bookkeeping stay outside that queue

## Implementation

### Step 1: patch the memory search manager

Locate `getMemorySearchManagerCacheStore()` and `getMemorySearchManager()` in the active `memory-*.js` bundle.

The patch must do all of the following:

- add `pendingQmdManagerCreates` to the cache store
- bind it as `PENDING_QMD_MANAGER_CREATES`
- keep the cached full-manager return and borrowed status-manager return before any pending-key computation
- move the create path into a local `createQmdManager(mode)` function
- use the pending map only for `mode === "full"`
- build the pending key from `baseCacheKey` plus `workspaceDir`, not the entire raw config
- for non-status full creates, store the pending QMD create promise in the map and clear it in `finally`
- if the awaited pending QMD create returns nothing, fall back via `getBuiltinMemorySearchManager(...)`
- clear `PENDING_QMD_MANAGER_CREATES` in `closeAllMemorySearchManagers()`

The live Bill shape now looks like this:

```ts
function getMemorySearchManagerCacheStore() {
  return resolveGlobalSingleton(MEMORY_SEARCH_MANAGER_CACHE_KEY, () => ({
    qmdManagerCache: new Map(),
    pendingQmdManagerCreates: new Map(),
  }));
}

const {
  qmdManagerCache: QMD_MANAGER_CACHE,
  pendingQmdManagerCreates: PENDING_QMD_MANAGER_CREATES,
} = getMemorySearchManagerCacheStore();
```

and inside `getMemorySearchManager(...)`:

```ts
const cached = QMD_MANAGER_CACHE.get(cacheKey);
if (cached) {
  return { manager: cached };
}
if (statusOnly) {
  const fullCached = QMD_MANAGER_CACHE.get(`${baseCacheKey}:full`);
  if (fullCached) {
    return { manager: new BorrowedMemoryManager(fullCached) };
  }
}

const workspaceDir = resolveAgentWorkspaceDir(params.cfg, params.agentId);
const createQmdManager = async (mode) => {
  ...
};

if (!statusOnly) {
  const pendingCreateKey = `${baseCacheKey}:${workspaceDir}`;
  const pending = PENDING_QMD_MANAGER_CREATES.get(pendingCreateKey);
  if (pending) {
    const manager = await pending;
    return manager ? { manager } : await getBuiltinMemorySearchManager(params);
  }

  const pendingCreate = createQmdManager("full").finally(() => {
    PENDING_QMD_MANAGER_CREATES.delete(pendingCreateKey);
  });
  PENDING_QMD_MANAGER_CREATES.set(pendingCreateKey, pendingCreate);
  const manager = await pendingCreate;
  return manager ? { manager } : await getBuiltinMemorySearchManager(params);
}

const manager = await createQmdManager("status");
return manager ? { manager } : await getBuiltinMemorySearchManager(params);
```

### Step 2: patch the QMD manager update critical section

Locate `runUpdate(...)` in the active `qmd-manager-*.js` bundle.

The patch must do all of the following:

- add `QMD_UPDATE_QUEUE_KEY = Symbol.for("openclaw.qmdUpdateQueueState")`
- add `getQmdUpdateQueueState()` returning `{ tails: new Map() }`
- add a `closeSignal` promise plus resolver on the manager instance
- resolve that `closeSignal` from `close()`
- add `withQmdUpdateQueue(task)` on the manager class, keyed by `this.qmdDir`
- change `runUpdate()` so only:
  - `exportSessions()`
  - `runQmdUpdateWithRetry(reason)`
  - `this.dirty = false`
  run inside that queue
- after the queued section, return early if `this.closed`
- leave embed work and `lastUpdateAt` / `docPathCache.clear()` outside the queue

The live Bill shape now looks like this:

```ts
const QMD_UPDATE_QUEUE_KEY = Symbol.for("openclaw.qmdUpdateQueueState");

function getQmdUpdateQueueState() {
  return resolveGlobalSingleton(QMD_UPDATE_QUEUE_KEY, () => ({ tails: new Map() }));
}
```

and inside `runUpdate(...)`:

```ts
const run = async () => {
  await this.withQmdUpdateQueue(async () => {
    if (this.closed) return;
    if (this.sessionExporter) await this.exportSessions();
    await this.runQmdUpdateWithRetry(reason);
    this.dirty = false;
  });

  if (this.closed) return;

  if (this.shouldRunEmbed(force)) {
    ...
  }

  if (this.closed) return;

  this.lastUpdateAt = Date.now();
  this.docPathCache.clear();
};
```

with helper:

```ts
async withQmdUpdateQueue(task) {
  const queue = getQmdUpdateQueueState();
  const key = this.qmdDir;
  const previous = queue.tails.get(key) ?? Promise.resolve();
  let releaseCurrent;
  const current = new Promise((resolve) => {
    releaseCurrent = resolve;
  });
  const next = previous.then(() => current, () => current);
  queue.tails.set(key, next);
  try {
    const waitResult = await Promise.race([
      previous.then(() => "ready", () => "ready"),
      this.closeSignal.then(() => "closed"),
    ]);
    if (waitResult === "closed") return;
    return await task();
  } finally {
    releaseCurrent();
    void next.finally(() => {
      if (queue.tails.get(key) === next) queue.tails.delete(key);
    });
  }
}
```

## Verification

After patching:

1. Restart the gateway.
2. Confirm the new bundle is loaded.
3. Verify the active `memory-*.js` contains `pendingQmdManagerCreates`.
4. Verify the active `memory-*.js` builds the pending-create key from effective QMD inputs plus `workspaceDir`, not the whole raw config blob.
5. Verify the active `qmd-manager-*.js` contains `QMD_UPDATE_QUEUE_KEY`, `closeSignal`, and `withQmdUpdateQueue`.
6. If QMD session export is enabled, watch for absence of the prior overlapping export/update failure pattern.
7. If a manager is closed while waiting behind another queued update, confirm shutdown is no longer delayed by the queued update window.

## When You Need This On Another Node

Another OpenClaw node needs OV-011 only if it is actually running the risky path:

- `memory.backend = "qmd"`
- `memory.qmd.sessions.enabled = true`

If a node is not using QMD session export, OV-011 is not relevant there.
