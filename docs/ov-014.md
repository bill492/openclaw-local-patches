# OV-014

## Purpose

OV-014 is the pathological session-export line wrapping patch for OpenClaw `2026.4.14`.

Use this when session export still keeps one message per output line even for pathological giant blobs, which makes downstream indexing fragile.

This document is the reusable operator reference. Another Codex session should be able to read only this file and implement the same patch on another OpenClaw node.

## What It Fixes

Normal session export intentionally keeps one line per message:

- `User: ...`
- `Assistant: ...`

That is fine for normal turns.

The problem is pathological long messages. Without wrapping, one toxic giant line can be exported as a single line and handed downstream to indexers or search systems.

OV-014 preserves the historical one-line-per-message shape for normal turns, but wraps pathological long exports into multiple lines while preserving source mapping.

## Why This Exists

This is a downstream robustness patch.

Without it:

- a giant user or assistant blob becomes one giant exported line
- downstream indexing and QMD-style consumers have to ingest that line whole
- line-oriented debugging and mapping become worse

With it:

- normal turns stay unchanged
- pathological turns are wrapped safely
- continuation lines still map back to the same original JSONL source line

## Exact Behavior Change

### Before

The export path did:

```ts
collected.push(`${label}: ${safe}`);
lineMap.push(jsonlIdx + 1);
messageTimestampsMs.push(parseSessionTimestampMs(record, message));
```

### After

The export path:

- defines `SESSION_EXPORT_WRAP_CHARS = 800`
- splits pathological long lines with `splitLongSessionLine(...)`
- renders wrapped output with `renderSessionExportLines(label, safe)`
- repeats `lineMap` entries for each wrapped continuation line
- repeats `messageTimestampsMs` entries for each wrapped continuation line

That last timestamp repetition is important on the current `2026.4.14` bundle shape because the export path now tracks `messageTimestampsMs` alongside `lineMap`.

## Implementation

Locate the session-files section in the active `engine-qmd-*.js` bundle.

Make these changes:

1. Add:

```ts
const SESSION_EXPORT_WRAP_CHARS = 800;
```

2. Add:

```ts
function splitLongSessionLine(text, maxChars = SESSION_EXPORT_WRAP_CHARS) {
  const normalized = text.trim();
  if (!normalized) return [];
  if (normalized.length <= maxChars) return [normalized];
  const segments = [];
  let cursor = 0;
  while (cursor < normalized.length) {
    const remaining = normalized.length - cursor;
    if (remaining <= maxChars) {
      segments.push(normalized.slice(cursor).trim());
      break;
    }
    const limit = cursor + maxChars;
    let splitAt = normalized.lastIndexOf(" ", limit);
    if (splitAt <= cursor) splitAt = limit;
    segments.push(normalized.slice(cursor, splitAt).trim());
    cursor = splitAt;
    while (cursor < normalized.length && normalized[cursor] === " ") cursor += 1;
  }
  return segments.filter(Boolean);
}

function renderSessionExportLines(label, text) {
  return splitLongSessionLine(text).map((segment) => `${label}: ${segment}`);
}
```

3. Replace:

```ts
collected.push(`${label}: ${safe}`);
lineMap.push(jsonlIdx + 1);
messageTimestampsMs.push(parseSessionTimestampMs(record, message));
```

with:

```ts
const renderedLines = renderSessionExportLines(label, safe);
collected.push(...renderedLines);
lineMap.push(...renderedLines.map(() => jsonlIdx + 1));
messageTimestampsMs.push(...renderedLines.map(() => parseSessionTimestampMs(record, message)));
```
