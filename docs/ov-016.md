# OV-016

## Purpose

OV-016 is the Slack thread-delivery patch note for two related seams:

- Part 1 is the generic outbound runtime sender fix that now matches the rebased upstream PR state.
- Part 2 is the separate Valhalla-local carry patch for parent-owned completion delivery.

Historically, both parts were carried locally on OpenClaw `2026.4.14`. That is no longer true in the same way:

- Part 1 should now be treated as the upstream-aligned behavior description and backport note.
- Part 2 should still be treated as a local carry patch unless and until it lands upstream separately.

Use this on another install when both of these are true:

- a subagent or routed Slack reply is supposed to stay in a Slack thread
- the reply can fall out to the channel or behave inconsistently because thread targeting is lost on the outbound path

This document is intentionally self-contained. Another Codex session should be able to read only this file and apply the same patch set.

## What It Fixes

OV-016 has two parts.

### Part 1: Preserve Slack thread aliases in the generic outbound runtime sender

OpenClaw's generic runtime outbound wrapper expects canonical fields:

- `messageThreadId`
- `replyToMessageId`

But Slack-originated sends can arrive with Slack alias fields instead:

- `threadTs`
- `replyToId`

If the wrapper forwards only the canonical fields, Slack thread context is dropped before the real sender is called. The visible symptom is a reply that should land in-thread posting at channel level instead.

The fix is to preserve both shapes at the wrapper boundary, with canonical fields winning when both are present.

On current upstream `main`, this landed in a narrower helper-based form than the original local carry patch:

- `replyToMessageId ?? replyToId` is already normalized through a local helper
- the remaining upstream delta is to add `threadTs` as a fallback in the generic runtime thread resolver
- this document should treat that helper-based upstream shape as the source of truth for Part 1

### Part 2: Route parent-owned completion delivery through the normal reply dispatcher

On Bill's Valhalla install, there is also a second active fix in the parent-owned subagent completion seam.

For `expectsCompletionMessage === true`, the direct announce path was using a raw `callGateway({ method: "agent", ... })` path instead of the normal routed reply dispatcher. That seam is now rerouted through the normal dispatcher so the parent-owned completion uses the same thread-aware reply machinery as ordinary threaded replies.

Carry Part 2 only if you want the exact same behavior Bill is currently running.

## Why This Exists

The practical problem is not "Slack threads are broken everywhere." The problem is narrower:

- some Slack replies go through provider-specific reply handling and work
- some Slack replies go through the generic runtime outbound sender and lose thread targeting unless alias fields are preserved
- parent-owned subagent completions can additionally bypass the normal routed reply path unless the direct announce seam is rerouted

OV-016 patches those exact seams without changing the higher-level ownership model.

Status summary:

- Part 1: upstream-aligned and suitable as a backport note or dist-patch translation guide
- Part 2: still a local carry patch

## Exact Behavior Change

### Part 1 behavior

Before:

```ts
type RuntimeSendOpts = {
  threadId?: string | number | null;
  messageThreadId?: string | number;
  replyToId?: string | number | null;
  replyToMessageId?: string | number;
  ...
};

function resolveRuntimeThreadId(opts: RuntimeSendOpts): string | number | undefined {
  return opts.messageThreadId ?? opts.threadId ?? undefined;
}

function resolveRuntimeReplyToId(opts: RuntimeSendOpts): string | undefined {
  const raw = opts.replyToMessageId ?? opts.replyToId;
  return raw == null ? undefined : normalizeOptionalString(String(raw));
}
```

After:

```ts
type RuntimeSendOpts = {
  threadId?: string | number | null;
  messageThreadId?: string | number;
  threadTs?: string | number;
  replyToId?: string | number | null;
  replyToMessageId?: string | number;
  ...
};

function resolveRuntimeThreadId(opts: RuntimeSendOpts): string | number | undefined {
  return opts.messageThreadId ?? opts.threadId ?? opts.threadTs ?? undefined;
}

function resolveRuntimeReplyToId(opts: RuntimeSendOpts): string | undefined {
  const raw = opts.replyToMessageId ?? opts.replyToId;
  return raw == null ? undefined : normalizeOptionalString(String(raw));
}
```

Contract:

- `messageThreadId` wins over `threadId`
- `threadId` wins over `threadTs`
- `replyToMessageId` wins over `replyToId`
- Slack alias fields are no longer dropped at the generic wrapper boundary

### Part 2 behavior

Before, the parent-owned completion direct-announce seam effectively did:

```ts
await subagentAnnounceDeliveryDeps.callGateway({
  method: "agent",
  params: {
    sessionKey: canonicalRequesterSessionKey,
    message: params.triggerMessage,
    deliver: deliveryTarget.deliver,
    ...
  },
  expectFinal: true,
  timeoutMs: announceTimeoutMs,
});
```

After, when all of the following are true:

- `params.expectsCompletionMessage`
- `deliveryTarget.deliver`
- `deliveryTarget.channel`
- `deliveryTarget.to`

the seam:

- creates a normal channel reply pipeline with `createChannelReplyPipeline(...)`
- builds a routed inter-session reply context carrying:
  - `OriginatingChannel`
  - `OriginatingTo`
  - `AccountId`
  - `MessageThreadId`
  - `ExplicitDeliverRoute: true`
- calls `dispatchReplyWithDispatcher(...)`
- treats success as `dispatchResult?.queuedFinal === true`
- returns an explicit failure if the dispatcher does not queue a final

This keeps the parent-owned completion on the same routed-delivery machinery ordinary threaded replies use.

## Implementation

### Step 1: patch the generic outbound runtime sender

Patch `createChannelOutboundRuntimeSend(...)` in the active `deps-*.js` bundle so it resolves Slack alias fields before calling the real sender.

There are now two equivalent shapes for Part 1, depending on the tree you are patching.

#### New upstream PR-aligned shape

On current upstream `main`, the clean target shape is helper-based. The effective change is:

- add `threadTs?: string | number` to `RuntimeSendOpts`
- extend `resolveRuntimeThreadId(...)` from `messageThreadId ?? threadId` to `messageThreadId ?? threadId ?? threadTs`
- leave `resolveRuntimeReplyToId(...)` in its local-hoisted form

That shape looks like this:

```ts
type RuntimeSendOpts = {
  ...
  threadId?: string | number | null;
  messageThreadId?: string | number;
  threadTs?: string | number;
  replyToId?: string | number | null;
  replyToMessageId?: string | number;
  ...
};

function resolveRuntimeThreadId(opts: RuntimeSendOpts): string | number | undefined {
  return opts.messageThreadId ?? opts.threadId ?? opts.threadTs ?? undefined;
}

function resolveRuntimeReplyToId(opts: RuntimeSendOpts): string | undefined {
  const raw = opts.replyToMessageId ?? opts.replyToId;
  return raw == null ? undefined : normalizeOptionalString(String(raw));
}
```

If the runtime sender already has these helpers, this is the preferred shape to carry because it matches the rebased upstream PR.

#### Older `2026.4.14` dist carry shape

On Bill's live `2026.4.14` bundle, the older dist seam did not yet have the same helper structure, so the carry patch may still need to resolve the values inline.

The old target shape is:

```ts
function createChannelOutboundRuntimeSend(params) {
  return { sendMessage: async (to, text, opts = {}) => {
    const outbound = await loadChannelOutboundAdapter(params.channelId);
    if (Boolean(opts.mediaUrl) && outbound?.sendMedia) return await outbound.sendMedia({
      ...
      threadId: opts.messageThreadId,
      replyToId: opts.replyToMessageId == null ? void 0 : normalizeOptionalString(String(opts.replyToMessageId)),
      ...
    });
    if (!outbound?.sendText) throw new Error(params.unavailableMessage);
    return await outbound.sendText({
      ...
      threadId: opts.messageThreadId,
      replyToId: opts.replyToMessageId == null ? void 0 : normalizeOptionalString(String(opts.replyToMessageId)),
      ...
    });
  } };
}
```

For that older bundle shape, replace it with the equivalent of:

```ts
function createChannelOutboundRuntimeSend(params) {
  return { sendMessage: async (to, text, opts = {}) => {
    const outbound = await loadChannelOutboundAdapter(params.channelId);
    const resolvedReplyTo = opts.replyToMessageId ?? opts.replyToId;
    const resolvedThreadId = opts.messageThreadId ?? opts.threadId ?? opts.threadTs;
    if (Boolean(opts.mediaUrl) && outbound?.sendMedia) return await outbound.sendMedia({
      ...
      threadId: resolvedThreadId,
      replyToId: resolvedReplyTo == null ? void 0 : normalizeOptionalString(String(resolvedReplyTo)),
      ...
    });
    if (!outbound?.sendText) throw new Error(params.unavailableMessage);
    return await outbound.sendText({
      ...
      threadId: resolvedThreadId,
      replyToId: resolvedReplyTo == null ? void 0 : normalizeOptionalString(String(resolvedReplyTo)),
      ...
    });
  } };
}
```

If you want an automated patch on an older bundle with the same inline shape, use this:

```bash
python3 - <<'PY'

    )
    if text2 != text:

```

If the local bundle shape differs, do not force a blind replacement. Find the current `createChannelOutboundRuntimeSend(...)` implementation and make the equivalent logical change for that tree:

- on helper-based trees: add `threadTs` and extend `resolveRuntimeThreadId(...)`
- on older inline trees: compute `resolvedReplyTo` and `resolvedThreadId`
- preserve precedence `messageThreadId -> threadId -> threadTs`
- preserve precedence `replyToMessageId -> replyToId`

### Step 2: patch the parent-owned direct-announce seam

Apply this only if you want the exact same Valhalla behavior Bill is carrying.

In the active `subagent-registry-*.js` bundle, locate `sendSubagentAnnounceDirectly(...)` and insert a routed dispatcher branch before the old raw `callGateway({ method: "agent", ... })` fallback.

Do not carry temporary trace logging into the patch. The behavior change is the dispatcher branch itself.

The branch condition must be:

```ts
params.expectsCompletionMessage &&
deliveryTarget.deliver &&
deliveryTarget.channel &&
deliveryTarget.to
```

Inside that branch, implement this behavior:

```ts
const agentId = resolveAgentIdFromSessionKey(canonicalRequesterSessionKey);
const { onModelSelected, ...replyPipeline } = createChannelReplyPipeline({
  cfg,
  agentId,
  channel: INTERNAL_MESSAGE_CHANNEL,
});

const routeCtx = {
  Body: params.triggerMessage,
  RawBody: params.triggerMessage,
  CommandBody: params.triggerMessage,
  InputProvenance: {
    kind: "inter_session",
    sourceSessionKey: params.sourceSessionKey,
    sourceChannel: params.sourceChannel ?? "webchat",
    sourceTool: params.sourceTool ?? "subagent_announce",
  },
  SessionKey: canonicalRequesterSessionKey,
  Provider: INTERNAL_MESSAGE_CHANNEL,
  Surface: INTERNAL_MESSAGE_CHANNEL,
  OriginatingChannel: deliveryTarget.channel,
  OriginatingTo: deliveryTarget.to,
  ExplicitDeliverRoute: true,
  AccountId: deliveryTarget.accountId,
  MessageThreadId: deliveryTarget.threadId,
  ChatType: "direct",
  CommandAuthorized: true,
  MessageSid: params.directIdempotencyKey,
};

const dispatchResult = await dispatchReplyWithDispatcher({
  ctx: routeCtx,
  cfg,
  dispatcherOptions: {
    ...replyPipeline,
    deliver: async () => {},
  },
  replyOptions: {
    runId: params.directIdempotencyKey,
    abortSignal: params.signal,
    onModelSelected,
  },
});

if (dispatchResult?.queuedFinal === true) {
  return { delivered: true, path: "direct" };
}

return {
  delivered: false,
  path: "direct",
  error: "routed-dispatch-did-not-queue-final",
};
```

Only after that branch should the old raw `callGateway({ method: "agent", ... })` path remain as fallback.
