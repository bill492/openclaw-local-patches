# OV-017

## Purpose

OV-017 is the Slack thread bootstrap context hygiene patch for OpenClaw `2026.4.16`.

Use this on another install when one or more of these are true:

- a fresh Slack thread session replays prior OpenClaw Slack bot replies back into the prompt
- Slack self-bot filtering compares `bot_id` thread authors against `auth.test().user_id`, so the match never fires
- retained third-party bot or integration replies are reformatted as `Unknown (user)`
- follow-up thread turns lose root-message context because `ThreadStarterBody` no longer has a fallback when `ThreadHistoryBody` is absent

This document is intentionally self-contained. Another Codex session should be able to read only this file and apply the same patch set.

## What It Fixes

OV-017 has three parts.

### Part 1: omit only the current Slack bot from new thread bootstrap

On a new Slack thread session, `resolveSlackThreadContextData(...)` can fetch existing thread replies and build `ThreadHistoryBody`.

Before the current OV-017 shape, that seeding path could replay OpenClaw's own earlier Slack bot replies back into the next fresh session. In captured cases, those replies included low-value commentary or reasoning-style text.

The current patch does not drop all bot-authored thread history. It drops only messages authored by the current Slack bot.

Matching must work across both Slack identifier domains:

- `auth.test().user_id` yields the bot user id, typically `U...`
- thread replies often identify bot authors via `bot_id`, typically `B...`

So a message counts as "current bot" when either:

- `author.userId === ctx.botUserId`
- `author.botId === ctx.botId`

### Part 2: preserve third-party bot context with bot-aware labeling

Retained third-party bot or integration replies should not be mislabeled as human-user text.

If a history entry survives the current-bot filter and it has `bot_id` but no `userId`, OV-017 keeps it in `ThreadHistoryBody` and formats it with a bot-aware sender label such as:

```text
Bot (B2) (assistant)
```

That preserves the bot identity and avoids degrading the entry into `Unknown (user)`.

### Part 3: restore thread-starter fallback only for follow-up turns

`runPreparedReply(...)` consumes `ThreadStarterBody` and `ThreadHistoryBody`.

The earlier OV-017 draft removed the plain-text `[Thread starter - for context]` fallback entirely. That solved new-session duplication, but it also created a follow-up-turn regression for channels that still rely on `ThreadStarterBody` without populating `ThreadHistoryBody`.

The current patch restores the starter fallback only when:

- `isNewSession === false`
- `ThreadHistoryBody` is absent
- `ThreadStarterBody` is present

That keeps new-session Slack thread bootstrap deduplicated while preserving root-message context on follow-up turns.

## Why This Exists

The problem is not "Slack thread bootstrap should never exist." The problem is narrower:

- new-session Slack thread seeding should preserve useful human and third-party context
- OpenClaw's own prior Slack bot replies should not be replayed into the next fresh session
- Slack self-bot filtering has to compare against both `user_id` and `bot_id`
- follow-up turns should still have a root-message fallback when only `ThreadStarterBody` is available

OV-017 patches those exact seams without changing the general first-turn Slack thread bootstrap model.

## Exact Behavior Change

### Part 1 behavior

Before, the Slack thread bootstrap path effectively treated every non-allowlisted omission the same, and it did not have a reliable "current bot" matcher across both Slack id domains.

On a matching source shape, it looked equivalent to:

```ts
const starterAllowed =
  !starter ||
  isSlackThreadContextSenderAllowed({
    allowFromLower: params.allowFromLower,
    allowNameMatching: params.allowNameMatching,
    userId: starter.userId,
    userName: starterSenderName,
    botId: starter.botId,
  });

const includeStarterContext =
  !starter ||
  shouldIncludeSupplementalContext({
    mode: params.contextVisibilityMode,
    kind: "thread",
    senderAllowed: starterAllowed,
  });

const uniqueUserIds = [
  ...new Set(threadHistory.map((item) => item.userId).filter((id): id is string => Boolean(id))),
];

const { items: filteredThreadHistory, omitted: omittedHistoryCount } =
  filterSupplementalContextItems({
    items: threadHistory,
    ...
  });
```

After, it should do the equivalent of:

```ts
const isCurrentBotAuthor = (author: { userId?: string; botId?: string }): boolean =>
  Boolean(
    (params.ctx.botUserId && author.userId && author.userId === params.ctx.botUserId) ||
    (params.ctx.botId && author.botId && author.botId === params.ctx.botId),
  );

const starterIsCurrentBot = Boolean(
  starter &&
    isCurrentBotAuthor({
      userId: starter.userId,
      botId: starter.botId,
    }),
);

const starterAllowed =
  !starter ||
  (!starterIsCurrentBot &&
    isSlackThreadContextSenderAllowed({
      allowFromLower: params.allowFromLower,
      allowNameMatching: params.allowNameMatching,
      userId: starter.userId,
      userName: starterSenderName,
      botId: starter.botId,
    }));

const includeStarterContext =
  !starter ||
  (!starterIsCurrentBot &&
    shouldIncludeSupplementalContext({
      mode: params.contextVisibilityMode,
      kind: "thread",
      senderAllowed: starterAllowed,
    }));

const threadHistoryWithoutCurrentBot = threadHistory.filter(
  (historyMsg) =>
    !isCurrentBotAuthor({
      userId: historyMsg.userId,
      botId: historyMsg.botId,
    }),
);
const omittedCurrentBotHistoryCount =
  threadHistory.length - threadHistoryWithoutCurrentBot.length;

const uniqueUserIds = [
  ...new Set(
    threadHistoryWithoutCurrentBot
      .map((item) => item.userId)
      .filter((id): id is string => Boolean(id)),
  ),
];

const { items: filteredThreadHistory, omitted: omittedHistoryCount } =
  filterSupplementalContextItems({
    items: threadHistoryWithoutCurrentBot,
    ...
  });
```

Contract:

- current-bot Slack starter text is omitted from context
- current-bot Slack history entries are not seeded into `ThreadHistoryBody`
- third-party bot or integration entries are still eligible to survive
- Slack self-bot filtering works whether Slack reports the author as `user` or `bot_id`

### Part 2 behavior

Before, retained history entries were formatted with bot-aware labels:

```ts
const msgSenderName =
  msgUser?.name ?? (historyMsg.botId ? `Bot (${historyMsg.botId})` : "Unknown");
const isBot = Boolean(historyMsg.botId);
const role = isBot ? "assistant" : "user";

from: `${msgSenderName} (${role})`,
```

The earlier OV-017 draft regressed this into:

```ts
const msgSenderName = msgUser?.name ?? "Unknown";

from: `${msgSenderName} (user)`,
```

The current OV-017 shape restores the bot-aware labeling:

```ts
const isBot = Boolean(historyMsg.botId);
const msgSenderName = msgUser?.name ?? (isBot ? `Bot (${historyMsg.botId})` : "Unknown");
const role = isBot ? "assistant" : "user";

from: `${msgSenderName} (${role})`,
```

Contract:

- retained third-party bot or integration messages keep bot identity
- `bot_id`-only Slack history entries are not mislabeled as `Unknown (user)`
- the current-bot omission logic still wins before formatting

### Part 3 behavior

Before the fix, `runPreparedReply(...)` could do this:

```ts
const threadStarterBody = normalizeOptionalString(ctx.ThreadStarterBody);
const threadHistoryBody = normalizeOptionalString(ctx.ThreadHistoryBody);
const threadContextNote = threadHistoryBody
  ? `[Thread history - for context]\n${threadHistoryBody}`
  : undefined;
```

That removed the thread-starter fallback completely.

After, it should do the equivalent of:

```ts
const threadStarterBody = normalizeOptionalString(ctx.ThreadStarterBody);
const threadHistoryBody = normalizeOptionalString(ctx.ThreadHistoryBody);
const threadContextNote = threadHistoryBody
  ? `[Thread history - for context]\n${threadHistoryBody}`
  : !isNewSession && threadStarterBody
    ? `[Thread starter - for context]\n${threadStarterBody}`
    : undefined;
```

Contract:

- new sessions still prefer `ThreadHistoryBody`
- new sessions do not get the duplicate plain-text starter fallback
- follow-up turns can still recover root thread context from `ThreadStarterBody` when history is absent
- if both `ThreadHistoryBody` and `ThreadStarterBody` are present, history wins

## Implementation

### Step 1: patch Slack provider bootstrap so context carries `bot_id`

Patch the active bundle containing:

```ts
const auth = await app.client.auth.test({ token: botToken });
botUserId = auth.user_id ?? "";
```

It should become equivalent to:

```ts
const auth = await app.client.auth.test({ token: botToken });
botUserId = auth.user_id ?? "";
botId = auth.bot_id ?? "";
```

And the Slack monitor context creation path must pass that `botId` through.

If you want another Codex session to patch this safely, the exact contract is:

- declare `let botId = ""`
- set it from `auth.test().bot_id`
- include it in the Slack monitor context object

### Step 2: patch Slack thread-context seeding in the active `prepare-*.js` bundle

Patch `resolveSlackThreadContextData(...)` so:

- it defines `isCurrentBotAuthor(...)`
- starter omission uses that helper
- history omission uses that helper
- `uniqueUserIds` is computed from the filtered history
- the omitted-history log includes both visibility omissions and current-bot omissions
- retained third-party bot entries still format with bot-aware sender labels

On a matching source shape, the logic should become equivalent to:

```ts
const isCurrentBotAuthor = (author) =>
  Boolean(
    (params.ctx.botUserId && author.userId && author.userId === params.ctx.botUserId) ||
    (params.ctx.botId && author.botId && author.botId === params.ctx.botId),
  );

const starterIsCurrentBot = Boolean(starter && isCurrentBotAuthor(...));

  (historyMsg) =>
    !isCurrentBotAuthor({
      userId: historyMsg.userId,

  threadHistory.length - threadHistoryWithoutCurrentBot.length;

...

from: `${msgSenderName} (${role})`,
```

If the local bundle shape differs, do not force a blind replacement. Find `resolveSlackThreadContextData(...)` and make the equivalent logical change described above.

### Step 3: patch `runPreparedReply(...)` in the active `reply-*.js` bundle

Locate the thread-context prelude logic containing:

```ts

const threadHistoryBody = normalizeOptionalString(ctx.ThreadHistoryBody);

  ? `[Thread history - for context]\n${threadHistoryBody}`
```

Replace it with:

