# OV-018

## Purpose

OV-018 is the subagent completion wake-vs-direct-delivery patch for OpenClaw `2026.4.19-beta.2`.

Use this when subagent completion announces still collapse two different cases into one rule:

- active requester sessions should be resumed internally
- dormant or manual requester sessions should still receive an externally deliverable completion message

This document is intentionally self-contained. Another Codex session should be able to read only this file and apply the same patch set on another install.

## What It Fixes

OV-018 fixes two runtime seams.

### Part 1: resolve requester session activity from the live embedded-run map

The stored requester-session entry can have a stale `sessionId`.

If completion delivery decides activity only from persisted store state, it can misclassify the requester as dormant even while there is an active embedded run for that session key.

OV-018 resolves activity from the live embedded-run session map first:

- `resolveActiveEmbeddedRunSessionId(requesterSessionKey)`
- then falls back to the persisted requester-session entry only when needed

### Part 2: wake active completion requesters internally, but keep dormant completion requesters externally deliverable

The broken intermediate behavior was:

- all completion announces were treated as session-internal
- direct delivery was marked handled too early
- dormant or manual completion requesters could lose the only externally visible completion message

OV-018 restores the split:

- if the requester session is active, wake it via `queueEmbeddedPiMessage(...)`
- if the requester session is not active, keep completion announces externally deliverable
- if the active wake cannot be injected, report the direct path as not delivered so the existing fallback path can still run

## Why This Exists

Subagent completion does not have one universal delivery rule.

When the requester session is actively resumable, the resumed requester turn should own the single user-visible reply. That avoids duplicate external completion messages.

When there is no active requester turn to resume, the completion announce itself must stay externally deliverable. Otherwise the completion can disappear entirely from the user-facing channel.

OV-018 patches that exact distinction without changing non-completion announce behavior.

## Exact Behavior Change

### Part 1 behavior

Before, requester activity was effectively derived from stored session-entry state:

```ts
const { cfg, entry } = loadRequesterSessionEntry(params.requesterSessionKey);
const canonicalKey = resolveRequesterStoreKey(cfg, params.requesterSessionKey);
const sessionId = entry?.sessionId;
if (!sessionId) {
  return "none";
}
const isActive = isEmbeddedPiRunActive(sessionId);
```

After, requester activity should be resolved from the live session map first:

```ts
const sessionId =
  resolveActiveEmbeddedRunSessionId(requesterSessionKey) ??
  loadRequesterSessionEntry(requesterSessionKey).entry?.sessionId;
return {
  sessionId,
  isActive: Boolean(sessionId && isEmbeddedPiRunActive(sessionId)),
};
```

and callers should use a helper equivalent to:

```ts
function resolveRequesterSessionActivity(requesterSessionKey: string) {
  const activity = subagentAnnounceDeliveryDeps.getRequesterSessionActivity(requesterSessionKey);
  if (activity.sessionId || activity.isActive) {
    return activity;
  }
  const { entry } = loadRequesterSessionEntry(requesterSessionKey);
  const sessionId = entry?.sessionId;
  return {
    sessionId,
    isActive: Boolean(sessionId && isEmbeddedPiRunActive(sessionId)),
  };
}
```

Contract:

- live requester activity wins over stale stored `sessionId`
- queue-steer decisions operate on the active embedded run when one exists
- stale requester-session store entries do not recreate the old duplicate/lost-delivery behavior

### Part 2 behavior

Before, completion-mode direct delivery was over-suppressed. The broken shape was equivalent to:

```ts
const shouldDeliverExternally = !params.expectsCompletionMessage && deliveryTarget.deliver;

return await callGateway({
  method: "agent",
  params: {
    ...,
    deliver: shouldDeliverExternally,
  },
});
```

That shape is wrong because it treats every completion announce as session-internal, even when there is no active requester turn to resume.

After, the direct-delivery path should behave like this:

```ts
const requesterActivity = resolveRequesterSessionActivity(canonicalRequesterSessionKey);
if (params.expectsCompletionMessage && requesterActivity.isActive) {
  const woke = requesterActivity.sessionId
    ? subagentAnnounceDeliveryDeps.queueEmbeddedPiMessage(
        requesterActivity.sessionId,
        params.triggerMessage,
      )
    : false;
  if (woke) {
    return {
      delivered: true,
      path: "steered",
    };
  }
  return {
    delivered: false,
    path: "direct",
    error: "active requester session could not be woken",
  };
}
```

Then continue using the normal direct external delivery path for:

- dormant completion requesters
- manual completion requesters with no active turn
- non-completion announces

Contract:

- active completion requesters are resumed internally through `queueEmbeddedPiMessage(...)`
- dormant completion requesters still get `deliver: true` external completion delivery when a route exists
- failed active wake attempts do not falsely count as successful direct delivery
- non-completion announce behavior stays externally deliverable

## Implementation

Locate the subagent announce delivery module and its runtime companion in the active dist bundle.

Make these changes.

### Step 1: re-export the live session resolver from the runtime shim

In the runtime bundle, extend the re-export from `./pi-embedded-runner/runs.js` so it also exports:

```ts
resolveActiveEmbeddedRunSessionId
```

### Step 2: add requester-activity seams to delivery deps

Extend `SubagentAnnounceDeliveryDeps` with:

```ts
getRequesterSessionActivity: (requesterSessionKey: string) => {
  sessionId?: string;
  isActive: boolean;
};
queueEmbeddedPiMessage: typeof queueEmbeddedPiMessage;
```

and set the default implementation to resolve activity from:

1. `resolveActiveEmbeddedRunSessionId(requesterSessionKey)`
2. then `loadRequesterSessionEntry(requesterSessionKey).entry?.sessionId`

### Step 3: add `resolveRequesterSessionActivity(...)`

Add a helper equivalent to:

```ts
function resolveRequesterSessionActivity(requesterSessionKey: string) {
  const activity = subagentAnnounceDeliveryDeps.getRequesterSessionActivity(requesterSessionKey);
  if (activity.sessionId || activity.isActive) {
    return activity;
  }
  const { entry } = loadRequesterSessionEntry(requesterSessionKey);
  const sessionId = entry?.sessionId;
  return {
    sessionId,
    isActive: Boolean(sessionId && isEmbeddedPiRunActive(sessionId)),
  };
}
```

### Step 4: use live requester activity in queue-mode announce handling

In `maybeQueueSubagentAnnounce(...)`, replace the direct `entry?.sessionId` lookup with:

```ts
const { sessionId, isActive } = resolveRequesterSessionActivity(canonicalKey);
```

and use the injected `subagentAnnounceDeliveryDeps.queueEmbeddedPiMessage(...)` when attempting to steer the active requester session.

### Step 5: split active completion wake from dormant completion direct delivery

In `sendSubagentAnnounceDirectly(...)`, before the normal external gateway delivery call, add logic equivalent to:

```ts
const requesterActivity = resolveRequesterSessionActivity(canonicalRequesterSessionKey);
if (params.expectsCompletionMessage && requesterActivity.isActive) {
  const woke = requesterActivity.sessionId
    ? subagentAnnounceDeliveryDeps.queueEmbeddedPiMessage(
        requesterActivity.sessionId,
        params.triggerMessage,
      )
    : false;
  if (woke) {
    return {
      delivered: true,
      path: "steered",
    };
  }
  return {
    delivered: false,
    path: "direct",
    error: "active requester session could not be woken",
  };
}
```

Do not suppress normal external delivery for dormant completion requesters. They should still reach the existing `deliver: true` gateway path when a direct route exists.

### Step 6: add regression coverage

Patch the matching tests so they cover:

- active requester completion announces wake the requester via `queueEmbeddedPiMessage(...)` and do not call the external gateway
- dormant completion requesters still use external direct delivery
- non-completion announces still use external direct delivery
- active wake failure falls back to queued follow-up delivery instead of double-sending

## Verification

After patching:

1. Restart the gateway.
2. Confirm the active delivery bundle contains:
   - `resolveActiveEmbeddedRunSessionId`
   - `resolveRequesterSessionActivity`
   - `active requester session could not be woken`
3. Verify an active requester-session completion:
   - wakes the requester internally
   - does not send a second external completion message
4. Verify a dormant or manual completion requester:
   - still receives an externally deliverable completion announce
5. Verify a non-completion announce:
   - still uses the normal external direct-delivery path
